CU-01

Ver receta aleatoria del día

Actor: Usuario Pantalla: HomeScreen Conexión requerida: Sí
✅ Precondición
  • El dispositivo tiene conexión activa a internet.
  • La aplicación CocinaApp está instalada y se está iniciando o ya está abierta.
  • La API TheMealDB está disponible.
🏁 Postcondición
  • Se muestra en HomeScreen una receta aleatoria con imagen, nombre y categoría.
  • El usuario puede acceder al detalle completo tocando la receta.
▶️ Flujo Ideal
  1. El usuario abre la aplicación CocinaApp.
  2. El sistema muestra un indicador de carga en HomeScreen.
  3. El sistema realiza una petición HTTP GET al endpoint /random.php de TheMealDB.
  4. La API responde con los datos de una receta aleatoria (nombre, imagen, categoría, área).
  5. El sistema procesa la respuesta y construye el modelo de receta.
  6. Se muestra la receta en HomeScreen: imagen de portada, nombre del platillo y categoría.
  7. El usuario puede tocar la receta para ver su detalle completo.
🔀 Flujos Alternativos
  • Alt-1: Recargar receta. Si el usuario toca el botón de recarga/actualizar, el sistema repite el flujo desde el paso 2 y muestra una nueva receta aleatoria.
  • Alt-2: Receta ya cargada. Si la aplicación ya estaba en segundo plano y se reactiva, puede mantener la receta cargada previamente sin hacer una nueva solicitud.
⚠️ Excepciones
  • EXC-1: Sin conexión a internet → Se muestra mensaje de error: "Sin conexión. Verifica tu red e intenta de nuevo." con botón de reintentar.
  • EXC-2: Tiempo de espera agotado (timeout) → Se muestra mensaje: "El servidor tardó demasiado. Intenta de nuevo."
  • EXC-3: API devuelve respuesta vacía o error 5xx → Se muestra mensaje de error genérico.
📋 Reglas
  • La receta debe ser aleatoria en cada apertura o recarga.
  • Siempre debe mostrarse un indicador de carga (spinner) mientras se espera la respuesta.
  • La imagen de la receta debe cargarse desde la URL provista por la API.
  • Si la imagen no carga, debe mostrarse una imagen de placeholder.
CU-02

Explorar categorías de recetas

Actor: Usuario Pantalla: HomeScreen / DrawerMenu Conexión requerida: Sí
✅ Precondición
  • El dispositivo tiene conexión a internet.
  • La aplicación está abierta en HomeScreen o el usuario abrió el DrawerMenu.
  • La API TheMealDB está disponible.
🏁 Postcondición
  • Se muestra la lista completa de categorías con su imagen y nombre.
  • El usuario puede seleccionar cualquier categoría para ver sus recetas.
▶️ Flujo Ideal
  1. El usuario abre la aplicación o desliza el Drawer lateral.
  2. El sistema realiza una petición HTTP GET al endpoint /categories.php.
  3. La API responde con la lista de categorías (nombre, imagen, descripción).
  4. El sistema construye la lista de modelos de categoría.
  5. Se muestran las categorías como chips o tarjetas en HomeScreen y como ítems en el DrawerMenu.
  6. El usuario puede desplazarse horizontalmente (en HomeScreen) o verticalmente (en Drawer) por las categorías.
🔀 Flujos Alternativos
  • Alt-1: El usuario llega a las categorías desde HomeScreen tocando la sección "Categorías destacadas".
  • Alt-2: El usuario llega a las categorías abriendo el DrawerMenu con el ícono de menú (hamburguesa) en la barra superior.
⚠️ Excepciones
  • EXC-1: Sin conexión → Mensaje: "No se pudieron cargar las categorías." con opción de reintentar.
  • EXC-2: Lista vacía en la respuesta → Mensaje: "No hay categorías disponibles en este momento."
  • EXC-3: Error en la API → Se muestra mensaje de error genérico.
📋 Reglas
  • Cada categoría debe mostrar su imagen y nombre.
  • La lista sigue el orden devuelto por la API.
  • Debe mostrarse indicador de carga mientras se obtienen las categorías.
  • La imagen de cada categoría debe tener un fallback si no carga.
CU-03

Filtrar recetas por categoría

Actor: Usuario Pantalla: CategoryScreen Conexión requerida: Sí
✅ Precondición
  • Las categorías están cargadas y visibles (CU-02 completado).
  • El usuario ha seleccionado una categoría específica.
  • El dispositivo tiene conexión a internet.
🏁 Postcondición
  • Se muestra CategoryScreen con un grid de recetas pertenecientes a la categoría seleccionada.
  • El título de la pantalla refleja el nombre de la categoría.
▶️ Flujo Ideal
  1. El usuario toca una categoría en HomeScreen o en el DrawerMenu.
  2. El sistema navega a CategoryScreen y muestra un indicador de carga.
  3. El sistema realiza una petición HTTP GET a /filter.php?c={nombre_categoría}.
  4. La API responde con la lista de recetas de esa categoría (id, nombre, imagen).
  5. El sistema construye los modelos de receta y los muestra en un grid responsivo.
  6. El usuario puede desplazarse por el grid y tocar cualquier receta para ver su detalle.
🔀 Flujos Alternativos
  • Alt-1: Volver atrás. El usuario presiona el botón "atrás" del sistema o el ícono ← en la barra superior para regresar a HomeScreen o al Drawer.
  • Alt-2: Cambiar de categoría. El usuario vuelve al Drawer y selecciona otra categoría; el sistema carga el nuevo filtro.
⚠️ Excepciones
  • EXC-1: Sin conexión → Mensaje de error con botón reintentar.
  • EXC-2: Categoría sin recetas → Mensaje: "No se encontraron recetas en esta categoría."
  • EXC-3: Nombre de categoría inválido o no reconocido por la API → Error genérico.
📋 Reglas
  • El grid debe ser responsivo: 2 columnas en celular, 3 en tablet (MediaQuery).
  • El nombre de la categoría debe mostrarse como título en la AppBar.
  • Cada tarjeta del grid muestra imagen y nombre de la receta.
  • Debe mostrarse indicador de carga durante la petición.
CU-04

Buscar receta por nombre

Actor: Usuario Pantalla: SearchScreen Conexión requerida: Sí
✅ Precondición
  • El dispositivo tiene conexión a internet.
  • El usuario ha navegado a la pestaña o pantalla de búsqueda (SearchScreen).
  • La API TheMealDB está disponible.
🏁 Postcondición
  • Se muestran los resultados de búsqueda que coincidan con el texto ingresado.
  • El usuario puede tocar un resultado para ver el detalle de esa receta.
▶️ Flujo Ideal
  1. El usuario navega a SearchScreen desde las tabs o el DrawerMenu.
  2. El sistema muestra un campo de texto (barra de búsqueda) vacío.
  3. El usuario escribe el nombre o parte del nombre de una receta.
  4. El sistema realiza una petición HTTP GET a /search.php?s={query} en tiempo real.
  5. La API devuelve la lista de recetas que coinciden con la búsqueda.
  6. El sistema muestra los resultados en una lista o grid con imagen y nombre.
  7. El usuario toca un resultado y navega a RecipeDetailScreen (CU-05).
🔀 Flujos Alternativos
  • Alt-1: Borrar búsqueda. El usuario borra el texto del campo → el sistema limpia los resultados y muestra el estado inicial de la pantalla.
  • Alt-2: Búsqueda sin resultados. La API responde con lista vacía → el sistema muestra mensaje de "sin resultados" (ver Excepciones).
  • Alt-3: Cancelar búsqueda. El usuario presiona atrás sin realizar búsqueda → regresa a la pantalla anterior.
⚠️ Excepciones
  • EXC-1: Sin conexión → Mensaje de error debajo del campo de búsqueda.
  • EXC-2: Sin resultados → Mensaje: "No se encontraron recetas para '[término buscado]'."
  • EXC-3: Query vacío → No se realiza ninguna solicitud a la API.
  • EXC-4: Error de red (timeout) → Mensaje de error con opción de reintentar.
📋 Reglas
  • La búsqueda se activa con al menos 1 carácter ingresado.
  • Debe mostrarse un indicador de carga mientras se espera respuesta.
  • Los resultados deben actualizarse automáticamente al escribir (tiempo real) o al presionar buscar.
  • El campo de búsqueda debe tener un botón para borrar el texto rápidamente (ícono ✕).
CU-05

Ver detalle completo de una receta

Actor: Usuario Pantalla: RecipeDetailScreen Conexión requerida: Preferente (no obligatoria si es favorita)
✅ Precondición
  • El usuario ha seleccionado una receta desde HomeScreen, CategoryScreen, SearchScreen o FavoritesScreen.
  • La receta tiene un ID válido para consultar el detalle.
  • Conexión a internet disponible o la receta está guardada en favoritos locales.
🏁 Postcondición
  • Se muestra RecipeDetailScreen con: imagen grande, nombre, categoría, área/origen, ingredientes con cantidades e instrucciones de preparación.
  • El botón de favorito muestra el estado actual (guardado / no guardado).
▶️ Flujo Ideal
  1. El usuario toca una receta en cualquier pantalla de listado.
  2. El sistema navega a RecipeDetailScreen y muestra un indicador de carga.
  3. El sistema realiza una petición HTTP GET a /lookup.php?i={idMeal}.
  4. La API devuelve el objeto completo de la receta con todos sus campos.
  5. El sistema procesa los ingredientes (strIngredient1..20) y medidas (strMeasure1..20), filtrando los vacíos.
  6. Se muestra la pantalla de detalle: imagen de portada a pantalla completa, nombre, categoría, área, lista de ingredientes con cantidades, instrucciones de preparación.
  7. Se muestra el ícono de favorito indicando si ya está guardada o no.
  8. El usuario puede tocar el botón de favorito para guardar/quitar la receta (CU-06 / CU-07).
🔀 Flujos Alternativos
  • Alt-1: Receta favorita sin conexión. Si la receta está guardada localmente, el sistema muestra los datos guardados sin hacer petición a la API.
  • Alt-2: Volver atrás. El usuario presiona ← para regresar a la pantalla de origen (HomeScreen, CategoryScreen, SearchScreen o FavoritesScreen).
⚠️ Excepciones
  • EXC-1: Sin conexión y sin datos locales → Mensaje: "Sin conexión. Esta receta no está disponible offline."
  • EXC-2: ID de receta inválido / no encontrado → Mensaje de error genérico.
  • EXC-3: La imagen de la receta no carga → Se muestra imagen de placeholder.
  • EXC-4: Timeout → Mensaje de error con botón de reintentar.
📋 Reglas
  • Los ingredientes deben listarse junto con su cantidad (strIngredient + strMeasure).
  • Solo se muestran ingredientes con valor no vacío (filtrar los 20 campos de la API).
  • Las instrucciones deben presentarse de forma legible, con saltos de párrafo.
  • El botón de favorito debe reflejar el estado real (guardado/no guardado) al cargar la pantalla.
CU-06

Guardar receta en favoritos

Actor: Usuario Pantalla: RecipeDetailScreen Conexión requerida: No (usa almacenamiento local)
✅ Precondición
  • El usuario está en RecipeDetailScreen con los datos de la receta cargados.
  • La receta no está guardada previamente en favoritos.
  • El almacenamiento local (SharedPreferences/SQLite) está disponible.
🏁 Postcondición
  • La receta queda guardada en el almacenamiento local del dispositivo.
  • El ícono de favorito (♡) cambia a su estado activo (♥).
  • La receta aparece en FavoritesScreen.
▶️ Flujo Ideal
  1. El usuario visualiza RecipeDetailScreen con el ícono de favorito en estado vacío (♡).
  2. El usuario toca el ícono de favorito.
  3. El sistema guarda los datos esenciales de la receta (id, nombre, imagen, categoría, ingredientes, instrucciones) en el almacenamiento local.
  4. El ícono de favorito cambia a estado activo (♥) con una animación de confirmación.
  5. El sistema muestra un Snackbar: "Receta guardada en favoritos ✓"
🔀 Flujos Alternativos
  • No existen flujos alternativos significativos para este caso de uso. Si la receta ya estaba guardada (por error de estado), el sistema la detecta y no la duplica.
⚠️ Excepciones
  • EXC-1: Error al escribir en el almacenamiento local → Mensaje: "No se pudo guardar la receta. Intenta de nuevo." El ícono no cambia de estado.
  • EXC-2: Almacenamiento lleno (poco probable) → Mensaje de error informativo.
📋 Reglas
  • Una misma receta no puede guardarse dos veces (no duplicados, validar por ID).
  • Los datos guardados deben ser completos para visualizar el detalle sin conexión.
  • El cambio de estado del ícono debe ser inmediato al confirmar el guardado.
  • La acción no requiere conexión a internet.
CU-07

Eliminar receta de favoritos

Actor: Usuario Pantalla: RecipeDetailScreen / FavoritesScreen Conexión requerida: No
✅ Precondición
  • La receta está guardada en el almacenamiento local (favoritos).
  • El usuario está en RecipeDetailScreen (ícono activo ♥) o en FavoritesScreen.
  • El almacenamiento local está disponible.
🏁 Postcondición
  • La receta se elimina del almacenamiento local.
  • El ícono de favorito vuelve a estado vacío (♡).
  • La receta desaparece de FavoritesScreen.
▶️ Flujo Ideal
  1. El usuario visualiza una receta con el ícono de favorito activo (♥) o la ve en FavoritesScreen.
  2. El usuario toca el ícono de favorito activo (desde DetailScreen) o realiza un gesto de deslizamiento/eliminación (desde FavoritesScreen).
  3. El sistema elimina los datos de la receta del almacenamiento local.
  4. El ícono cambia a estado vacío (♡).
  5. El sistema muestra un Snackbar: "Receta eliminada de favoritos" con opción de Deshacer.
  6. Si el usuario toca "Deshacer" dentro del tiempo límite, el sistema vuelve a guardar la receta (CU-06).
🔀 Flujos Alternativos
  • Alt-1: Eliminación desde FavoritesScreen. El usuario desliza la tarjeta de la receta hacia la derecha/izquierda → aparece confirmación o se elimina directamente con opción de Deshacer.
  • Alt-2: El usuario presiona "Deshacer". El sistema restaura la receta en favoritos y el ícono vuelve al estado activo (♥).
⚠️ Excepciones
  • EXC-1: Error al eliminar del almacenamiento local → Mensaje: "No se pudo eliminar la receta." El ícono no cambia.
  • EXC-2: La receta ya no existe en favoritos (estado inconsistente) → El sistema sincroniza el ícono al estado correcto sin mostrar error al usuario.
📋 Reglas
  • La eliminación debe reflejarse inmediatamente en todas las pantallas.
  • El cambio de estado del ícono debe ocurrir antes de mostrar el Snackbar.
  • La opción "Deshacer" debe estar disponible por al menos 3 segundos.
  • La acción no requiere conexión a internet.
CU-08

Ver recetas favoritas sin conexión

Actor: Usuario Pantalla: FavoritesScreen Conexión requerida: No
✅ Precondición
  • El usuario tiene al menos una receta guardada en el almacenamiento local.
  • El almacenamiento local está accesible.
  • El dispositivo puede o no tener conexión a internet.
🏁 Postcondición
  • Se muestra la lista de recetas favoritas guardadas localmente.
  • El usuario puede consultar el detalle de cualquier receta guardada sin necesitar internet.
▶️ Flujo Ideal
  1. El usuario navega a FavoritesScreen desde las tabs o el DrawerMenu.
  2. El sistema lee la lista de recetas guardadas desde el almacenamiento local.
  3. Se muestra la lista de recetas favoritas con imagen y nombre de cada una.
  4. El usuario toca una receta para ver su detalle completo (datos locales).
  5. RecipeDetailScreen muestra la información guardada previamente.
🔀 Flujos Alternativos
  • Alt-1: Con conexión a internet. Al tocar una receta favorita, el sistema puede recargar el detalle desde la API para obtener la información más actualizada.
  • Alt-2: Eliminar favorito. El usuario elimina una receta desde FavoritesScreen (CU-07); la lista se actualiza inmediatamente.
⚠️ Excepciones
  • EXC-1: Sin recetas guardadas → Mensaje: "Aún no tienes recetas favoritas. Explora y guarda las que más te gusten." con ilustración o ícono.
  • EXC-2: Error al leer el almacenamiento local → Mensaje de error con opción de reintentar.
📋 Reglas
  • La pantalla debe funcionar completamente sin conexión a internet.
  • Deben mostrarse todos los datos guardados: imagen, nombre, ingredientes e instrucciones.
  • La lista debe actualizarse automáticamente al agregar o eliminar favoritos.
  • Debe indicarse el número total de recetas guardadas (ej. "3 recetas guardadas").
CU-09

Navegar entre secciones con el Drawer

Actor: Usuario Pantalla: DrawerMenu (todas las pantallas) Conexión requerida: Parcial (para categorías)
✅ Precondición
  • La aplicación está abierta en cualquier pantalla que tenga acceso al Drawer.
  • Para mostrar la lista de categorías en el Drawer, se requiere conexión a internet.
🏁 Postcondición
  • El usuario es navegado a la sección o categoría seleccionada.
  • El Drawer se cierra automáticamente al seleccionar una opción.
  • La opción activa (pantalla actual) se muestra visualmente resaltada.
▶️ Flujo Ideal
  1. El usuario desliza el dedo desde el borde izquierdo de la pantalla o toca el ícono ☰ en la AppBar.
  2. El sistema abre el Drawer lateral con animación.
  3. Se muestran las opciones principales: Inicio, Buscar, Favoritos; y la lista de categorías con imágenes.
  4. El usuario selecciona una opción de navegación.
  5. El sistema cierra el Drawer con animación y navega a la pantalla o categoría seleccionada.
  6. Si seleccionó una categoría, el sistema ejecuta el flujo CU-03.
🔀 Flujos Alternativos
  • Alt-1: Cerrar sin seleccionar. El usuario toca fuera del Drawer o desliza hacia la izquierda → el Drawer se cierra sin navegar.
  • Alt-2: Seleccionar pantalla actual. El usuario toca la opción de la pantalla en la que ya se encuentra → el Drawer se cierra sin cambiar de pantalla.
  • Alt-3: Sin conexión. El Drawer muestra las opciones principales (Inicio, Buscar, Favoritos) pero la sección de categorías muestra un mensaje de error.
⚠️ Excepciones
  • EXC-1: Las categorías no cargan por falta de conexión → Se muestra mensaje dentro del Drawer: "No se pudieron cargar las categorías." Las opciones principales siguen funcionando.
  • EXC-2: Error inesperado al navegar → El sistema permanece en la pantalla actual y muestra un mensaje de error.
📋 Reglas
  • La sección activa (pantalla actual) debe mostrarse visualmente resaltada en el Drawer.
  • Las categorías en el Drawer deben mostrar imagen e ícono junto al nombre.
  • Las opciones principales (Inicio, Buscar, Favoritos) siempre deben estar disponibles, con o sin conexión.
  • El Drawer debe cerrarse automáticamente al seleccionar cualquier opción.
